SLIDE 3 — OUR SOLUTION: RECTITUDE.AI
================================================================================

Title: Rectitude.AI — A 5-Layer Security Sandwich for LLM Applications

Content:
Rectitude.AI is a production-grade, multi-layered security gateway that 
wraps LLM inference in 5 defensive tiers, implementing a NIST-aligned 
"Defense-in-Depth" strategy.

Core Differentiators:
    1. Not a single-point filter — 5 independent defense layers
    2. Not just input scanning — monitors behavior over multi-turn sessions
    3. Not static rules — RL-based adaptive policy that learns from attacks
    4. Not trust-based — cryptographic capability tokens for every tool call
    5. Not reactive — automated red teaming that proactively finds vulnerabilities

Architecture Philosophy:
    → Every prompt passes through the Security Sandwich before reaching the LLM
    → Every LLM response passes through the Output Mediator before reaching the user
    → Every tool call requires a cryptographically signed Capability Token
    → Every session is continuously scored by the Agent Stability Index (ASI)

SLIDE 5 — LAYER 1: INTENT SECURITY (THE SHIELD)
================================================================================

Title: Layer 1 — Intent & Prompt Security

Technique:    Hybrid Detection Pipeline (Regex → ML → Perplexity)
Latency:      <5ms (Tier 1) to ~150ms (Tier 2)
Accuracy:     95.4% (DeBERTa v3), 62% (Regex alone), 99%+ (combined)

Sub-Components:

    A) Regex Prefilter (regex_prefilter.py — 218 lines)
       • 40+ compiled regex patterns across 13 threat categories
       • Categories: role-switch, instruction override, system prompt extraction,
         jailbreak personas, indirect injection, encoding tricks, tool injection,
         data exfiltration, privilege escalation, social engineering, NSFW content,
         file probing, persistent injection
       • Scoring: Each pattern has a base_score (0.60 to 1.0)
       • Scores ≥1.0 = instant block (no ML needed)
       • Fusion: max(max_score, accumulated * 0.7) → fused decision

    B) DeBERTa v3 Classifier (injection_classifier.py — 94 lines)
       • Model: protectai/deberta-v3-base-prompt-injection-v2
       • Trained specifically for prompt injection (not content moderation)
       • Threshold: ≥0.80 → block, ≥0.50 → escalate
       • Lazy-loaded, GPU-accelerated when available

    C) GPT-2 Perplexity Detector (perplexity_detector.py — 97 lines)
       • Uses GPT-2 to compute token-level perplexity
       • Normal prompts: perplexity ~50-300
       • Obfuscated/encoded: perplexity ~5000+
       • Blocks base64, rot13, unicode-escaped attack payloads
       • Adaptive thresholds for short prompts

    D) Policy Engine (policy_engine.py — 165 lines)
       • Weighted signal fusion across all detectors:
           regex_prefilter : 0.40 weight
           injection_ml    : 0.35 weight
           perplexity      : 0.15 weight
           behavioral_asi  : 0.10 weight
       • Live threshold reads from PolicyStore (RL-updatable)
       • Custom patterns from RL agent applied as pre-pass
       • ML correction: ML can "vote down" regex false positives

Models Used:
    • protectai/deberta-v3-base-prompt-injection-v2 (HuggingFace)
    • gpt2 (HuggingFace) for perplexity scoring
    • Custom regex engine (hand-tuned 40+ patterns)

SLIDE 6 — LAYER 2: CRYPTOGRAPHIC INTEGRITY (THE VAULT)
================================================================================

Title: Layer 2 — Capability Tokens & Cryptographic Tool Control

Technique:    HMAC-signed JWT Capability Tokens
TTL:          300 seconds (5 minutes)
Differentiator: No competitor (Lakera, Giskard) implements this

How It Works:
    1. When an agent needs to call a tool, the orchestrator issues a 
       Capability Token scoped to specific tools.
    2. The token is signed with HMAC-SHA256 using the server's secret key.
    3. Before any tool executes, it verifies the token signature and scope.
    4. The LLM cannot forge these tokens — cryptographic proof of authorization.

Token Payload:
    {
        "sub":   "session_id",
        "scope": ["read_file", "search_web"],   // only these tools allowed
        "jti":   "unique-uuid",
        "exp":   "5 minutes from now",
        "iat":   "issued timestamp"
    }

Risk-Based Scope:
    • risk_score < 0.25 → Full scope (all 15 tools)
    • risk_score 0.25-0.50 → No destructive tools (read-only)
    • risk_score ≥ 0.50 → Minimal scope (read_file, search_web, read_crm only)

Known Tools (15 total):
    read_file, write_file, delete_file, search_web, send_email, 
    read_crm, log_ticket, query_database, execute_code, read_calendar,
    create_task, calculator, fetch_stock_price, get_current_time

High-Risk Tools (require explicit grant):
    delete_file, send_email, execute_code, query_database

Additional: Simulated FHE (Fully Homomorphic Encryption) engine — 
placeholder for future privacy-preserving inference when concrete-ml stabilizes.

SLIDE 7 — LAYER 3: BEHAVIORAL MONITORING (THE WATCHER)
================================================================================

Title: Layer 3 — Agent Stability Index (ASI) — Proprietary Multi-Turn Monitor

Technique:    Sliding Window Behavioral Drift Detection
Window Size:  10 interactions
Key Innovation: Detects "slow-burn" attacks that single-turn filters miss

ASI Formula:
    ┌─────────────────────────────────────────────────┐
    │  ASI = 0.35·C + 0.30·T + 0.20·B + 0.15·(1-LDFR) │
    └─────────────────────────────────────────────────┘

Where:
    C  (Consistency)   = Average cosine similarity between consecutive prompts
                         (bag-of-words, floor at 0.30)
                         Weight: 0.35

    T  (Tool Stability) = Penalizes sessions that introduce new tools late 
                           in the window (late tool introduction = suspicious)
                           Penalty: -0.25 per new late tool
                           Weight: 0.30

    B  (Boundary)       = Token count variance (coefficient of variation)
                          + persistence-after-block penalty (×0.6 if ≥2 blocks)
                          Weight: 0.20

    LDFR (Flip Rate)    = Fraction of consecutive pairs with similarity <0.15
                          (sharp topic flips ≈ social engineering)
                          Weight: 0.15

Thresholds:
    ASI > 0.80   → Green  (Stable Session)
    ASI 0.55-0.80 → Yellow (Warning)
    ASI < 0.55   → Red    (Alert — 2 consecutive windows below threshold)
                   → Forces ML scan on all subsequent prompts
                   → Revokes capability tokens

State Storage:
    Primary:  Redis (SETEX with 1-hour TTL)
    Fallback: In-process dictionary (for demo/dev mode)

Title: Layer 4 — Automated Red Teaming with Reinforcement Learning

Technique:    PPO (Proximal Policy Optimization) via Stable-Baselines3
Dataset:      JailbreakBench + 22 Built-in Attack Catalogue
Key Result:   0.0% Attack Success Rate on built-in catalogue

A) Attack Runner (attack_runner.py — 269 lines)
    • 22 built-in attacks across 8 categories:
        override (3), persona (3), extraction (3), indirect (2),
        exfil (3), privilege (2), multi_turn (4), encoding (2)
    • 5 legitimate prompts for false positive testing
    • JailbreakBench dataset integration (parquet loader)
    • Outputs: vulnerability_report.json

B) RL Policy Updater (rl_policy_updater.py — 737 lines)

    Environment: RedTeamEnv (Gymnasium)
        State Space (11 floats): [asr, fpr, n_patterns_norm, 
            block_thresh_norm, escalate_thresh_norm,
            bypass_override, bypass_exfil, bypass_extraction,
            bypass_persona, bypass_encoding, bypass_multi_turn]
        
        Action Space (Discrete 8):
            0: no-op
            1: add_suggested_pattern
            2: tighten_block_threshold (-0.05)
            3: relax_block_threshold (+0.05)
            4: tighten_escalate_threshold (-0.05)
            5: relax_escalate_threshold (+0.05)
            6: remove_worst_pattern
            7: reset_thresholds

        Reward Function:
            +2.0 × (ASR reduction)
            -1.0 × (FPR increase)
            -0.1 for no-op
            +0.5 bonus for ASR < 10%
            +0.2 bonus for FPR < 5%

    PPO Hyperparameters:
        learning_rate:  3e-4
        n_steps:        256
        batch_size:     64
        n_epochs:       10
        gamma:          0.99
        gae_lambda:     0.95
        clip_range:     0.2
        ent_coef:       0.01

    Pipeline:
        train(n_steps) → propose_updates(report) → interactive_review() → apply_approved()
        
    Live Policy Update: Changes written to PolicyStore → policy_engine.py reads 
    on next request → no server restart needed

SLIDE 9 — LAYER 5: ORCHESTRATION & OUTPUT MEDIATION
================================================================================

Title: Layer 5 — Intelligent Orchestration & Data Loss Prevention

A) Orchestrator (orchestrator.py — 319 lines)
    Risk-Tiered Routing:
        Tier 1 (<5ms):   Regex prefilter — blocks obvious attacks instantly
        Tier 2 (<200ms): ML classifiers — parallel async, for ambiguous requests
        Tier 3:          Capability token — for tool-calling requests
        Behavioral:      ASI update — always, non-blocking

    Routing Logic:
        prefilter ≥ 0.85       → Block immediately, skip ML
        prefilter 0.50–0.84    → Run ML classifiers (parallel)
        prefilter < 0.50       → Fast-pass with light ML scan
        tool_calls present     → Always run capability token check
        ASI alert active       → Force ML regardless of prefilter

B) Output Mediator (output_mediator.py — 109 lines)
    Scans LLM responses for data leakage:
        [CRITICAL] OpenAI API keys (sk-...), Anthropic keys, GitHub tokens, AWS keys
        [HIGH]     SSNs, credit cards, bearer tokens, salary figures, credentials
        [MEDIUM]   Phone numbers
        [LOW]      Email addresses (logged but not blocked)

    Actions:
        • Redacts all findings in audit copy
        • Blocks response only for medium+ severity
        • Returns sanitized response to user

C) Multi-Agent System (4 Specialized Agents):
    ┌──────────────┬─────────────────────┬────────────────────────────┐
    │ Agent        │ Domain              │ Security Sandbox           │
