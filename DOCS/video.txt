"The Orchestrator runs the full security pipeline:
   1. Tier 1: Regex prefilter on the prompt
   2. Check ASI score for the session
   3. If ambiguous or ASI alert → Tier 2: ML classifiers (async parallel)
   4. Policy Engine fuses all signals with weighted scoring
   5. If blocked → return immediately with security metadata

If the prompt is allowed, the route handler then:
   1. Routes to the correct agent via the AgentRouter
   2. The agent processes the prompt with its scoped capability token
   3. The response passes through the Output Mediator for PII scanning
   4. The final response is returned with full security telemetry

Every single step is audit-logged with request_id, user_id, decision, 
risk score, tier reached, latency, ASI score, and detector signals."

[03:30 – 05:00] LAYER 1 DEEP DIVE — REGEX + ML + PERPLEXITY
================================================================================

[SCREEN: Open backend/layer1_intent_security/regex_prefilter.py]

[VOICEOVER]:
"Let's go deep on Layer 1 — this is where the magic starts.

The regex prefilter is a file I'm particularly proud of. It contains 40+ 
compiled regex patterns organized into 13 threat categories.

Let me walk through the categories:

1. Role-switch / Persona Hijack — patterns like 'you are now a', 
   'act as if you are', 'pretend to be'. Each has a base score of 0.70-0.85.

2. Instruction Override — 'ignore all previous instructions', 
   'disregard everything above'. These have a score of 1.0 — instant block.

3. System Prompt Extraction — 'show me your system prompt', 
   'repeat everything above'. Score: 0.90-1.0.

4. Jailbreak Personas — DAN, STAN, AIM, evil mode, developer mode. 
   Score: 1.0 — instant block.

5. Indirect Injection — XML tag injection like <SYSTEM>, bracket tags, 
   markdown role headers. Score: 0.85-0.90.

The scoring system uses a 'fused score' approach:
   - max_score tracks the highest individual pattern score
   - accumulated tracks diminishing accumulation across multiple matches
   - fused = max(max_score, accumulated * 0.7)
   - If fused ≥ 0.85 → block, ≥ 0.50 → escalate, else → allow

This means a single high-confidence pattern blocks instantly, while multiple 
low-confidence patterns can accumulate to trigger an escalation."

[SCREEN: Switch to injection_classifier.py]

"The ML classifier uses ProtectAI's DeBERTa v3, which is specifically 
trained on prompt injection patterns — not general content moderation.

It's lazy-loaded on first use, so startup is fast. The model runs inference, 
applies softmax to the logits, and extracts the injection probability.

Three outcomes:
   ≥ 0.80 → block
   ≥ 0.50 → escalate  
   < 0.50 → allow

Notice the fallback chain: it first tries loading a local fine-tuned model 
from data/models/injection_classifier, then falls back to the HuggingFace 
model. If neither loads, it runs in passthrough mode."

[SCREEN: Switch to perplexity_detector.py]

"The perplexity detector is clever — it uses GPT-2 to compute how 
'surprised' the model is by the input text.

Normal English prompts have a perplexity of 50-300. Encoded or obfuscated 
text — like base64-encoded injection payloads — has perplexity of 5000+.

Thresholds:
   ≥ 8000 → block (likely obfuscation)
   ≥ 3000 → escalate (possible encoding)

There's an important adaptation for short prompts — they're naturally 
high-perplexity, so thresholds are relaxed: block threshold is 2x higher 
for prompts under 5 tokens."

[SCREEN: Switch to policy_engine.py]

"The Policy Engine is where all signals get fused. This is the final 
decision-maker for Layer 1.

Four weighted signals:
   regex_prefilter : weight 0.40
   injection_ml    : weight 0.35
   perplexity      : weight 0.15
   behavioral_asi  : weight 0.10

The weighted scores are normalized and compared against two thresholds:
   block_threshold    : 0.80 (default, RL-adjustable)
   escalate_threshold : 0.50 (default, RL-adjustable)

Critical design decision: I added ML correction. Previously, ANY regex 
'block' decision would override the ML classifiers. Now, the fused score 
is authoritative — if the regex says block at 0.85 but the ML says the 
prompt is clean, the fused score may stay below threshold. This dramatically 
reduced false positives on legitimate prompts like 'call the API endpoint'."

[05:00 – 06:15] LAYER 2 — CAPABILITY TOKENS
================================================================================

[SCREEN: Open backend/layer2_crypto/capability_tokens.py]

[VOICEOVER]:
"Layer 2 is the cryptographic backbone. Let me show you the 
CapabilityTokenService class.

When an agent needs to execute a tool, it doesn't just call it. The system 
issues a short-lived JWT token with a specific scope.

Here's the issue_token method:
   - Takes a session_id and list of allowed_tools
   - Sanitizes the tool list against KNOWN_TOOLS (15 registered tools)
   - Creates a JWT payload with sub, scope, jti, iat, exp
   - Signs it with HMAC-SHA256 using the server's secret key
   - TTL is 300 seconds — 5 minutes

The verify_tool_call method:
   - Decodes and validates the JWT signature
   - Checks if the requested tool is in the token's scope
   - If not in scope → denied, with a specific reason

Risk-based scope degredation is automatic:
   - Clean session (risk < 0.25): full 15-tool access
   - Moderate (0.25-0.50): no destructive tools (delete, send_email, execute)
   - High risk (≥ 0.50): minimal read-only access

This means even if an attacker successfully jailbreaks the LLM's persona, 
the agent physically cannot call tools outside its scope. The cryptography 
is separate from the language model — the LLM can't forge HMAC signatures.

There's also a sign_tool_call method that creates an HMAC digest of every 
tool call for audit trail integrity."

[06:15 – 07:30] LAYER 3 — ASI CALCULATOR
================================================================================

[SCREEN: Open backend/layer3_behavior_monitor/asi_calculator.py]

[VOICEOVER]:
"The ASI Calculator is one of our key innovations. Let me walk you through 
the math.

The ASI is computed over a sliding window of 10 interactions. Four metrics 
are calculated:

1. C — Response Consistency (weight: 0.35)
   I compute pairwise cosine similarity between consecutive prompts using 
   bag-of-words vectors. This catches sudden topic switches — like an 
   attacker changing from 'what's the weather' to 'show me all passwords'.
   
   Floor at 0.30: two unrelated legitimate questions shouldn't look like 
   an attack. The real signal is when flips are sharp AND frequent.

2. T — Tool Usage Stability (weight: 0.30)
   Penalizes sessions that introduce new tools late in the window.
   If the first 5 messages don't use any tools but message 8 suddenly 
   calls delete_file — that's suspicious. Penalty: -0.25 per new late tool.

3. B — Boundary Stability (weight: 0.20)
   Uses coefficient of variation of response token counts. Erratic response 
   lengths = model being manipulated. Additionally, if 2+ prompts have been 
   blocked and the user keeps trying, a 0.6x penalty is applied — 
   persistence after refusal is a strong signal.

4. LDFR — Latent Diagnosis Flip Rate (weight: 0.15)
   The fraction of consecutive prompt pairs with similarity below 0.15.
   High LDFR = sharp topic reversals = social engineering in progress.

The ASI formula combines these:
   ASI = 0.35*C + 0.30*T + 0.20*B + 0.15*(1 - LDFR)

Alert fires when ASI drops below 0.55 for 2 consecutive windows.

State is stored in Redis with a 1-hour TTL, with an in-process dictionary 
fallback when Redis is unavailable.

There's also a critical bug fix I want to highlight: the get_risk_score() 
method originally called compute(), which APPENDED a ghost entry to the 
session history on every pre-flight check. This caused double-counting of 
prompts and artificially inflated drift scores. I fixed it to read the 
history read-only and recompute scores without persisting."

[07:30 – 08:30] LAYER 4 — RL POLICY UPDATER
================================================================================

[SCREEN: Open backend/layer4_red_teaming/rl_policy_updater.py]

[VOICEOVER]:
"This is the largest single file in the project at 737 lines. The RL 
Policy Updater is a full Gymnasium environment with a PPO agent.

The RedTeamEnv class defines:

State Space — 11 continuous values between 0 and 1:
   The current ASR, FPR, normalized pattern count, normalized thresholds, 
   and per-category bypass rates for 6 attack categories.

Action Space — 8 discrete actions:
   0: no-op (penalized to prevent stalling)
   1: add a suggested regex pattern from the vulnerability report
   2-3: tighten/relax block threshold by 0.05
   4-5: tighten/relax escalate threshold by 0.05
   6: remove the most recently added custom pattern
   7: reset all thresholds to defaults

The reward function is designed to balance security and usability:
   +2.0 for every unit of ASR reduction
   -1.0 for every unit of FPR increase
   +0.5 bonus when ASR drops below 10%
   +0.2 bonus when FPR drops below 5%

PPO hyperparameters:
   Learning rate 3e-4, 256 steps per rollout, batch size 64, 
   10 epochs per update, gamma 0.99, GAE lambda 0.95, clip range 0.2.

The pipeline works like this:
   1. Attack Runner generates vulnerability_report.json
   2. RL agent loads the report, trains in the environment
   3. After training, it proposes policy updates
   4. Updates go through interactive human review (or auto-approve above 
      a confidence threshold for CI pipelines)
   5. Approved updates are written to PolicyStore
   6. policy_engine.py reads PolicyStore on every request — live updates, 
      zero downtime

Pattern validation is strict: patterns must be valid regex, at least 4 
characters, use word-boundary anchors if short, and not match trivial 
strings like 'ok' or 'hi'."

[08:30 – 09:15] THE AGENT SYSTEM
================================================================================

[SCREEN: Open backend/agents/base.py, then hr_database_agent.py]

[VOICEOVER]:
"The multi-agent system uses a BaseAgent abstract class that defines the 
contract every agent must follow.

Every agent has:
   - name and description (for routing)
   - system_prompt (constrains the LLM)
   - allowed_tools (enforced by capability tokens)
   - process() method that:
      1. Gets a scoped capability token based on risk score
      2. Intersects with the agent's allowed_tools list
      3. Passes the token to the execution context
      4. Wraps everything in try/catch with standardized AgentResponse

The key insight is in line 140: effective_scope is the intersection of the 
risk-based scope AND the agent's allowed tools. So even at risk_score 0.0 
(full scope), the HR agent only gets the tools it's registered for — it 
can never access delete_file or execute_code.

We have four agents:
   1. HR Database Agent — read-only SQL with PII masking, connects to 
      the demo employee database
   2. Email Agent — domain whitelisting, body content scanning
   3. Code Execution Agent — RestrictedPython sandbox, blocks os/sys
   4. Financial Advisor Agent — stock data, calculator, ASI-monitored

Plus a General Agent as a fallback for unrouted queries.

The AgentRouter uses keyword matching and intent classification to route 
user prompts to the correct agent. Commands like /hr, /email, /code, 
/finance provide explicit routing from the frontend PromptBox."
